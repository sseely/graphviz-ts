## Mechanism (irreducible)

The input file contains a naked UTF-8 trail byte `0x80` inside a node name.
C treats naked trail bytes 0x80–0xBF as "valid characters representing
themselves" (`lib/common/utils.c:1200-1207`, no warning emitted for this
case), and node-name `<title>` text never passes through charset conversion
(`agnameof` bytes go straight to `gvputs_xml`). The oracle therefore emits
the raw byte, producing an SVG that is declared UTF-8 but is not valid UTF-8.

The port's I/O boundary is JS strings (browser library; no byte-buffer
pipeline). Invalid-UTF-8 input is decoded with the latin1 fallback
(`0x80 → U+0080`) and re-encoded as valid UTF-8 on output. A JS string
cannot carry "the raw invalid byte 0x80" through `renderSvg`'s string
return value, so byte-matching C here would require corrupting the output
encoding for every consumer. This is a constraint below the code
(representation layer), per diagnosis.md stop condition 2.

## Disposition

Accepted as class **A5 — invalid input bytes (encoding representation)**.
The port's latin1 fallback matches C's own "treated as Latin-1" recovery
semantics for label text; only the raw-byte pass-through in names is
unreproducible. Re-measure only if the corpus input is fixed upstream.
