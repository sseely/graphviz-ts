# T1 — Enumerate + classify the dot corpus

## Context

graphviz-ts is a faithful TS port of C Graphviz (`~/git/graphviz`, tag 15.0.0 —
the spec). This mission builds a differential parity **survey** over the real
graphviz dot corpus; T1 produces the input inventory the survey consumes.

The corpus lives under `~/git/graphviz/tests` (configurable root). Sample facts
(2026-06-19): `tests/graphs/` holds 256 `.gv` inputs, all dot-targetable, none
with a `layout=` engine override; 130 use records/html/ports/clusters. Other
subdirs (`tests/`, `share/`, `nshare/`, `windows/`, `linux.*`, `macosx/`)
contain more `.gv`/`.dot`.

## Task

`enumerate` the corpus and `identify all` inputs, classifying each as
**applicable** (dot-targetable) or **quarantined** with an explicit reason from
the taxonomy in [../decisions.md](../decisions.md#quarantine-taxonomy). Write a
small `enumerate.ts` that produces `corpus-manifest.json` and a `README.md`.

1. `enumerate.ts`: walk the corpus root (default `~/git/graphviz/tests`, overridable
   via arg/env), collect `*.gv` and `*.dot` files. For each, read the head and
   classify:
   - **applicable** — parses as a single DOT graph, default engine (no
     `layout=` attr selecting another engine).
   - **quarantined** — one of: `engine-deferred` (has `layout=neato|fdp|sfdp|
     circo|twopi|osage` or lives in an engine-specific dir), `gvpr`,
     `multi-graph` (>1 top-level graph), `include`/`non-graph`,
     `raster-only-ref`, `parse-unsupported`.
   - Do the cheap structural checks here (regex/string), NOT a full render —
     rendering is T2. `parse-unsupported` may be left for T2 to discover and
     fold back; in T1, only mark inputs that are obviously non-DOT.
2. Emit `corpus-manifest.json`: an array of
   `{ id, path, engine: "dot", status: "applicable"|"quarantined",
   reason?: string }`. `id` = stable slug from the relative path.
3. `README.md`: how to run the harness (T1/T2/T3), corpus root/cache env vars,
   and what each status/reason means.

## Interface contract (consumed by T2)

```jsonc
// corpus-manifest.json
[{ "id": "graphs-abstract", "path": "graphs/abstract.gv",
   "engine": "dot", "status": "applicable" },
 { "id": "graphs-clust",   "path": "graphs/clust.gv",
   "engine": "dot", "status": "quarantined", "reason": "multi-graph" }]
```
`path` is relative to the corpus root. T2 reads only `status == "applicable"`.

## Acceptance criteria (Given/When/Then)

- **Given** the corpus root, **when** `enumerate.ts` runs, **then**
  `corpus-manifest.json` classifies **every** discovered `.gv`/`.dot` file —
  none unclassified.
- **Given** an input with `layout=neato`, **when** enumerated, **then** its
  status is `quarantined` / reason `engine-deferred`.
- **Given** the applicable set, **when** inspected, **then** every entry is a
  default-engine DOT graph (spot-check ≥5 by hand against the source).
- **Given** the manifest, **when** counted, **then** the README reports the
  applicable vs quarantined totals (per reason).
