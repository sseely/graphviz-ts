# Deep case: 1489

- **Corpus path:** `1489.dot` · **engine:** `dot`
- **firstDiffPath:** `svg/g[1]/g[1]/text[1]/text()[1]`
- **Port:** oracle unavailable (native dot error) — parser rejects: `Expected … but "?" found`
- **Oracle:** oracle unavailable (native dot error)
- **Root-cause group:** G5 — binary / corrupt input + charset
- **Why deep:** Non-ISO extended-ASCII file with bytes 0x80–0xFF scattered throughout, plus null
  bytes (`\x00`) and NEL (0x85) line terminators. C's lexer processes raw bytes as NAME chars.
  Even with latin1 re-decode, null bytes and binary content remain, requiring a tolerant lexer
  mode in addition to encoding handling.
- **Follow-on bucket:** `charset-encoding`

Deferred per ADR-3 (exceeds the localized ≤~30-line / single-module cutoff;
needs charset-aware reading AND tolerant lexer mode). Not fixed in this mission.
