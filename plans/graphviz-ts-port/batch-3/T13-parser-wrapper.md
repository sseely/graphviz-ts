# T13 — Parser TypeScript Wrapper and Tests

## Context

T12 produces `src/parser/dot.pegjs` and the Peggy build step produces
`src/parser/dot.js` and `src/parser/dot.d.ts`. T13 wraps these generated
artifacts in a TypeScript API that converts the raw Peggy AST into the
`Graph` model objects defined in `src/model/` (Batch 1).

The TypeScript API surface follows the DOT Parser Options section of
`~/git/graphviz/docs/architecture/typescript-port.md`. The `parse` function
is the primary entry point for in-memory DOT source. The `parseFile` function
is Node.js-only and must not be included in the browser bundle (guard with
a runtime environment check or place in a separate `node` export condition).

A `ParseError` class wraps Peggy's error type and exposes `line` and `column`
for diagnostic messages.

**Round-trip fidelity** is the key correctness criterion: parsing a `.dot`
file, serializing the resulting `Graph` to DOT text, and re-parsing must
produce a `Graph` with identical attribute values. Node and edge counts must
also be preserved. This is stronger than parse-only correctness.

**Edge direction validation**: The wrapper (not the grammar) validates that
`->` edges only appear in `digraph` documents and `--` edges only in `graph`
documents. Throw `ParseError` with a useful message if this is violated.

**Attribute value coercion**: DOT attribute values are always strings at the
model level (`Agsym_t.defval` in `lib/cgraph` is `char *`). The wrapper
stores all attribute values as `string`, never `number` or `boolean`, even
if the value looks numeric. This matches the C behavior.

   ```typescript

   // In Node environments, import { parseFile } from 'graphviz-ts/node'
   export async function parseFile(path: string): Promise<Graph>;
   ```

   The `Graph` type is imported from `src/model/index.ts` (Batch 1).

   The conversion from Peggy AST to `Graph`:
   - Create a `Graph` with `strict`, `directed` (digraph = true), and
     optional `id` attributes.
   - Walk `stmts` recursively: create nodes, edges, subgraphs.
   - Apply attribute statements (`graph [...]`, `node [...]`, `edge [...]`)
     as defaults on the current graph/subgraph scope.
   - Node attribute lists `[k=v, ...]` set per-node attributes.
   - Edge attribute lists apply to all edges in the edge chain.

2. **`src/parser/parser.test.ts`**: Tests.

   Tests must cover:
   - All `.dot` fixture files (see acceptance criteria)
   - Round-trip: parse → serialize → re-parse → compare node count, edge
     count, and all attribute values
   - `ParseError` with `line` and `column` for malformed input
   - HTML label attribute values preserved verbatim
   - `strict digraph` sets `graph.strict = true` and `graph.directed = true`
   - Edge direction validation: `->` in undirected graph throws `ParseError`
   - Attribute string coercion: numeric-looking values stored as string

   For the fixture files, use a small set of known-good `.dot` files. Create
   three fixtures in `src/parser/__fixtures__/`:
   - `simple.dot`: a 3-node 2-edge undirected graph with string attributes

   - `subgraph.dot`: a graph with two subgraph clusters and nested attributes

   Do not depend on `~/git/graphviz/graphs/` at test time (those are system
   files, not committed to the repo). The acceptance criterion requiring
   those files is validated manually by the implementer, not in automated

## Interface Contracts

```typescript
// src/parser/index.ts

import type { Graph } from '../model/index.js';

export class ParseError extends Error {
  readonly line: number;
  readonly column: number;
}

export function parse(src: string): Graph;
export async function parseFile(path: string): Promise<Graph>;
// parseFile available in Node only; see exports map in package.json
```

`Graph` from `src/model/` must have at minimum:
- `strict: boolean`
- `directed: boolean`
- `id: string | null`
- `nodes: Map<string, Node>`
- `edges: Edge[]`
- `subgraphs: Graph[]`
- `attrs: Map<string, string>`      // graph-level attributes
- `nodeDefaults: Map<string, string>` // node default attributes
- `edgeDefaults: Map<string, string>` // edge default attributes

If `src/model/graph.ts` (Batch 1) does not already have all these fields,
add them in `src/model/` as part of this task (declare the write to
`src/model/graph.ts` in the commit).

## Acceptance Criteria

**Given** any `.dot` file in `~/git/graphviz/graphs/` (verified manually),  
**When** `parse(src)` is called with its contents,  
**Then** node count and edge count match the C `agread` result for that file.

**Given** a round-trip: `parse(src)` → `serialize(graph)` → `parse(output)`,  
**When** both parse results are compared,  
**Then** node counts, edge counts, and all attribute values are identical
(string equality; no coercion).

**Given** a DOT source with an invalid statement (e.g., unclosed brace),  
**When** `parse(src)` is called,  
**Then** a `ParseError` is thrown with `line >= 1` and `column >= 1`.
