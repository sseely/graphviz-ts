# T0.2 — Re-bucket the not-cleared set + refined bucket list

## Context
After T0.1, `attribution-sfdp.jsonl` holds the fresh drift-exonerated /
not-cleared split. The not-cleared ids are the real tracked residual; group
them into the mission's buckets so Batches 1-5 target real work (and skip
buckets emptied by the regen).

## Task
Read the fresh not-cleared set; for each id compute (a) firstDiff object-kind
(graph-bb vs edge), (b) input attributes (ratio=fill, splines=ortho, record,
rankdir), (c) platform-family membership. Emit `batch-0/findings.md`: one row
per DISTINCT input (representative), its platform copies, its bucket (B1-B5),
and a one-line mechanism hypothesis. Flag any bucket now empty.

## Read-set
- `test/corpus/attribution-sfdp.jsonl` (from T0.1)
- `test/corpus/parity-sfdp.jsonl` (firstDiff per id)
- `~/git/graphviz/tests/<id>` inputs (attribute scan)
- the bucket table in `../README.md`

## Write-set
- `batch-0/findings.md`

## Interface output (consumed by Batches 1-5)
```
findings.md table columns:
  representative: string   (e.g. "graphs-unix")
  copies: string[]         (platform-duplicate ids sharing the input)
  bucket: "B1".."B5"
  firstDiffKind: "graph-bb" | "edge"
  attrs: string[]          (ratio=fill | ortho | record | rankdir | plain)
  hypothesis: string       (one-line mechanism guess)
```

## Acceptance criteria
- Given the not-cleared set, when bucketed, then every id appears in exactly one
  representative row (as representative or copy).
- Given a bucket with no surviving not-cleared ids, when writing findings, then
  it is explicitly marked EMPTY (its batch will be skipped).
- Given findings.md, when read by a later batch, then the representative id and
  bucket are unambiguous (no id in two buckets).
