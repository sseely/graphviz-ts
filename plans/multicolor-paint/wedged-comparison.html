Wedged node fill — port (V8) vs C graphviz 15.0.0 (glibc)

Known divergence (libm stop-condition, not a port bug). Wedged fills render correctly, but the elliptic-arc subdivision count differs: the port's estimateError lands at ~4.5e-6 in V8 vs >1e-5 in glibc, straddling the 1e-5 threshold — so the port emits fewer bezier segments per wedge than C (the "coord-pairs/path" counts below). The wedge geometry is visually equivalent; only the control points (and thus conformant parity with the glibc-generated refs) differ. The wedged golden is therefore excluded, not tolerance-pinned (a segment-count difference is structural, not within numeric tolerance).
