# T6 — sfdp Attribution + Bucket Analysis

## Context

`test/corpus/parity-sfdp.json` (2026-07-11 sweep): 762 total, 260
pass, 494 diverged, 7 oracle-error, 1 port-error, 0 timeout, tolerance
0.5. sfdp's stochastic/multilevel layout (see `.agent-notes/sfdp-*`
and the memory index's "sfdp oracle + FP stability" entry) is
pinnable to 6 digits with a matched PRNG — read that precedent before
assuming any drift here is unfixable jitter.

## Task

1. Run `test/corpus/attribute-divergence.ts sfdp` (T1, batch-1) to
   completion across the full diverged set.
2. Run `test/corpus/oracle-error-classifier.ts sfdp` (T3, batch-1) on
   the 7 `oracle-error` rows. Separately triage the 1 `port-error` row
   by hand (it is not an oracle failure — read its `err` field in
   `parity-sfdp.json` directly) and record its cause in
   `sfdp-buckets.md`; it does not go through T3's classifier.
3. Write `batch-2/sfdp-buckets.md`: rank buckets by member count as in
   T4. Given sfdp's multilevel coarsening, expect (but verify, don't
   assume) that some buckets correlate with graph size — note in the
   doc whether large-graph ids cluster in one bucket vs. being spread
   evenly, since that's diagnostic of a coarsening-stage vs.
   final-refinement-stage mechanism.
4. Do not fix anything in this task — attribution and bucketing only.
