# T5 — fdp Attribution + Bucket Analysis

## Context

`test/corpus/parity-fdp.json` (2026-07-11 sweep): 762 total, 311 pass,
435 diverged, 15 oracle-error, 0 port-error, 1 timeout, tolerance 0.5.
fdp has a known standing lead for part of its tail: `clusteredges.c`
(compound cluster-endpoint edge routing) is unported —
`plans/port-catalog/README.md:358`. This task's bucket analysis should
actively check whether that gap accounts for a bucket, since it would
let batch-3 skip straight to a scoped port rather than re-diagnosing
from scratch.

## Task

1. Run `test/corpus/attribute-divergence.ts fdp` (T1, batch-1) to
   completion across the full diverged set.
2. Run `test/corpus/oracle-error-classifier.ts fdp` (T3, batch-1) on
   the 15 `oracle-error` + 1 `timeout` rows.
3. Write `batch-2/fdp-buckets.md`: rank buckets by member count as in
   T4. Additionally: for the largest not-cleared (non-drift-exonerated)
   bucket(s), check whether the corpus ids involved use compound edges
   into/out of a cluster (grep the corpus `.gv`/`.dot` source for
   `cluster` + inspect whether the diverging edges cross a cluster
   boundary). If so, label that bucket `clusteredges-gap` explicitly in
   the doc and cite `plans/port-catalog/README.md:358` — this becomes
   batch-3's first-round target for fdp.
4. Do not fix anything in this task — attribution and bucketing only.
