# T0.1 — Capture C's edge dispatch order + order-sensitive reads

## Context
The port routes edges in two passes (groups, then lone); C routes both in one
`edgecmp`-ordered loop (`lib/dotgen/dotsplines.c:dot_splines_`, the
`make_regular_edge`/`make_flat_edge` dispatch ~line 400-420). The order matters
because `recover_slack` mutates chain vnode coords and `top_bound`/`bot_bound`
read neighbors' installed splines. We must capture C's exact order before
changing the port.

## Task
1. Add `getenv("GV_DUMP_MRE")`-gated dumps to `~/git/graphviz/lib/dotgen/dotsplines.c`:
   - At the `make_regular_edge`/`make_flat_edge` dispatch (~line 410-419): print
     each edge group's `agnameof(agtail(e0))->agnameof(aghead(e0)) cnt=N` in
     dispatch order (a `[MRE-ORDER]` line per group).
   - In `recover_slack` (~line 2054): print each vnode it moves
     (`name rank y x_before->x_after box=[ll,ur] label`) (`[MRE-SLACK]`).
2. Rebuild only the dot-layout plugin: `cd ~/git/graphviz/build &&
   make gvplugin_dot_layout` (symlinked into `/tmp/ghl`; recipe in
   `[[recover-slack-and-c-harness]]`).
3. Render `graphs/ldbxtried.gv` with the instrumented oracle
   (`GV_DUMP_MRE=1 GVBINDIR=/tmp/ghl ~/git/graphviz/build/cmd/dot/dot -Tsvg`);
   record: the full dispatch order, the position of `n0->n1` (expect ~30) vs the
   `n0->n2` group (expect ~85, cnt=3, virtual head `%0`), and the recover_slack
   move of the rank-1 vnode `%0` (expect 967->789).
4. Build a **minimal synthetic repro**: a lone edge whose corridor reads a chain
   vnode that a parallel/opposing group (routed LATER in `edgecmp`) repositions.
   Capture its dispatch order + the move. Keep it as small as possible.
5. Record everything in `comparisons/c-order-oracle.md`: the order list, the
   move(s), and explicitly which edges read which moved vnode and whether C routes
   them before or after the move.
