# T0.2 — Port order divergence + edgecmp containment check

## Context
The port routes parallel/opposing groups in `splines.ts:dotSplines_` (its
`routeEdgeGroup` loop over `collectNodeEdges`-collected, `edgecmp`-sorted edges;
`dispatchEdgeGroup` early-returns lone edges), then routes lone edges in a second
pass `edge-route.ts:routeDotEdges`. We need the port's actual order vs C's
(T0.1), and — critically — whether the port's `edgecmp` (the order
`dotSplines_` already uses) reproduces C's dispatch order. ADR-5 GO requires it.

## Task
1. Add temporary `DBG_ORDER`-gated stderr dumps:
   - In `dotSplines_`'s `routeEdgeGroup` loop: print each group `tail->head cnt`
     in the order processed (pass 1).
   - In `routeDotEdges`/`routeOneEdge`: print each lone edge `tail->head` in the
     order processed (pass 2).
2. Render `ldbxtried` + the T0.1 repro; collect both passes' order.
3. **Containment check:** compare the port's full edge order (pass 1 groups +
   pass 2 lone, AND the underlying `edgecmp` order from `collectNodeEdges`) to
   C's dispatch order from `c-order-oracle.md`. Determine: does the port's
   `edgecmp` (the single sort `dotSplines_` already computes) — when NOT split
   into two passes — reproduce C's order exactly? Where do lone edges fall in
   `edgecmp` relative to groups?
4. **Remove the instrumentation**; confirm `git diff src/` is empty after this task.
5. Produce `comparisons/port-vs-c-order.md`: side-by-side order tables, the
   two-pass divergence (which lone edges C routes before a group that the port
   routes after), and a clear verdict: `edgecmp == C order` (GO-eligible) or
   `edgecmp diverges` (scope the alignment / STOP).
