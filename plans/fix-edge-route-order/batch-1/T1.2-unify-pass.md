# T1.2 — Unify into one edgecmp-ordered pass (Option A)

## Context
Core fix (ADR-3). `dotSplines_` collects all edges (`collectNodeEdges`), sorts by
`edgecmp`, and loops `routeEdgeGroup` — but `dispatchEdgeGroup` early-returns lone
edges (`uniq.length<=1`), deferring them to a SECOND pass `routeDotEdges`. C
routes lone and group edges in ONE `edgecmp` loop, so a lone edge sees neighbors'
splines and `recover_slack`-moved vnodes exactly as C does. Implement the single
pass per T0.3.

## Task (refined by T0.3 root-cause)
1. In `dotSplines_`'s `routeEdgeGroup`/`dispatchEdgeGroup` (`splines.ts`): when a
   group is lone (`cnt==1` after dedup), route it IN PLACE via the lone-edge
   dispatch (`routeOneEdge`) instead of returning early. Keep `cnt>1` →
   `routeParallelEdgeGroup`. Preserve flat-edge handling.
2. In `edge-route.ts`: remove the separate `routeDotEdges` invocation
   (`splines.ts:~488`) — its per-edge logic (`routeOneEdge`: dir!=forward, adjacent
   back, multi-rank, regular) now runs from the unified loop at each lone edge's
   `edgecmp` position. Ensure `routeOneEdge` is callable mid-loop (no two-pass
   assumptions, e.g. global "all groups done" state).
3. Preserve `cnt==1` byte-output for graphs with NO lone/group shared-state
   interaction (most of the 395 conformant graphs): the only behavioral change is
   the ORDER lone edges route relative to groups.
4. Make the T1.1 goldens pass; keep the diff minimal and within the write-set.
